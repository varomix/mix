C Backend & IRIX Guide

Compile MIX programs to portable C for any platform with a C compiler — including SGI IRIX on MIPS.

01Overview

MIX ships with two compilation backends:

BackendFlagPipelinePlatforms
LLVM (default) none .mix → LLVM IR → clang → binary Native targets supported by clang; also wasm32 and wasm-browser
C --backend c .mix → C source → cc → binary Any platform with a C compiler

The C backend translates MIX directly to portable C99 code, bypassing LLVM entirely. The generated C is then compiled by whatever cc is available on the target system — MIPSpro on IRIX, GCC on Linux, Clang on macOS.

LLVM backend (default)
.mix → AST → lower/LIR → .ll → clang → binary

C backend (--backend c)
.mix → AST → c_emit → .c → cc → binary

02Usage

Basic Commands

# Build with C backend
mix --backend c build game.mix

# Build and run
mix --backend c run game.mix

# Inspect generated C (no compilation)
mix --backend c --emit-ir -o game.c game.mix

# Set backend via environment variable
export MIX_BACKEND=c
mix build game.mix       # uses C backend automatically
mix run game.mixshell

Generated Output

Use --emit-ir with the C backend to inspect the generated C code:

mix --backend c --emit-ir -o output.c hello.mixshell

This produces a self-contained .c file with:

  • Standard #include headers (stdio.h, stdint.h, math.h, etc.)
  • extern declarations for all MIX runtime functions
  • Shape struct typedef definitions
  • Forward declarations for all MIX functions
  • Function and method implementations

Example

Given this MIX program:

shape Vec2
    x, y: float

    length() -> float
        sqrt(x * x + y * y)

main()
    v = Vec2(x: 3.0, y: 4.0)
    print(v.length())mix

The C backend generates:

Value shapes lower to ordinary C structs and locals. Explicit heap- or zone-owned forms such as Box[T] and collections remain pointer-based runtime values.

/* Generated by mix --backend c */
#include <stdio.h>
#include <stdint.h>
#include <math.h>

extern void mix_print_float(double);
/* ... more runtime declarations ... */

typedef struct {
    double x;
    double y;
} Vec2;

static double Vec2_length(Vec2 *self) {
    double t0 = self->x;
    double t1 = self->x;
    double t2 = t0 * t1;
    double t3 = self->y;
    double t4 = self->y;
    double t5 = t3 * t4;
    double t6 = t2 + t5;
    double t7 = mix_math_sqrt(t6);
    return t7;
}

int main(int argc, char **argv) {
    mix_set_args((int32_t)argc, argv);
    Vec2 _shape_8;
    Vec2 *t8 = &_shape_8;
    memset(t8, 0, sizeof(Vec2));
    double t9 = 3;
    t8->x = (double)t9;
    double t10 = 4;
    t8->y = (double)t10;
    Vec2 v_v;
    memcpy(&v_v, t8, sizeof(Vec2));
    Vec2 *t11 = &v_v;
    double t12 = Vec2_length((Vec2 *)t11);
    mix_print_float(t12);
    return 0;
}C

03Building for SGI IRIX

The C backend was specifically designed to target SGI IRIX on MIPS hardware (R10000/R12000/R14000 processors). IRIX has MIPSpro cc but no modern LLVM toolchain — the C backend solves this.

Prerequisites

  • SGI Octane 2 (or any IRIX workstation) running IRIX 6.5
  • MIPSpro C compiler (cc) — included with IRIX IDO
  • POSIX pthreads (included with IRIX)

Step 1: Build the MIX Compiler on IRIX

Transfer the MIX source tree to your IRIX machine and build the compiler natively:

# On the SGI Octane 2 (IRIX 6.5)
cd /usr/people/yourname/mix_lang

# Build with MIPSpro cc in 64-bit mode
# Note: override CC since Makefile defaults to cc,
# and add -64 for 64-bit MIPS ABI
make clean
make CC="cc -64" CFLAGS="-O2 -64"shell
Note: The LLVM backend is intended for the actively supported native and WebAssembly targets. On IRIX, use --backend c when compiling MIX programs.

Step 2: Compile MIX Programs

# Compile a MIX program using the C backend
./build/mix --backend c build game.mix

# Or set the environment variable
export MIX_BACKEND=c
./build/mix build game.mix
./build/mix run game.mix

# Install system-wide
make install PREFIX=/usr/local CC="cc -64" CFLAGS="-O2 -64"

# Then use from anywhere
mix --backend c run hello.mixshell

Step 3: Conditional Compilation

Use @os and @arch directives to write platform-specific code:

// Platform-specific paths
@os == "irix"
    get_temp_dir() -> str
        "/var/tmp"

@os == "macos"
    get_temp_dir() -> str
        "/tmp"

// Architecture-specific code
@arch == "mips64"
    print_arch~()
        print("Running on MIPS64")

@arch == "aarch64"
    print_arch~()
        print("Running on ARM64")mix

Supported values:

DirectiveValues
@os"macos", "linux", "irix"
@arch"aarch64" / "arm64", "x86_64", "mips64", "mips"
@debug(boolean, true when --debug flag is set)
@release(boolean, true when --debug is not set)

IRIX-Specific Notes

64-bit mode: Always compile with -64 on IRIX. MIX assumes 64-bit pointers (sizeof(void*) == 8). The default IRIX ABI is 32-bit (-n32), which will cause pointer truncation.
  • MIPSpro flags: The generated C is C99-compatible. MIPSpro supports C99 with -c99 flag. If you encounter issues, try CC="cc -64 -c99".
  • Math library: The linker automatically includes -lm for math functions (sqrt, sin, cos, etc.).
  • Pthreads: MIX concurrency (go/wait/shared) uses POSIX pthreads, which IRIX supports natively. No extra flags needed.
  • Runtime: The MIX runtime (lib/runtime.c) is portable C and compiles on IRIX without modification.

Cross-Compilation

You can also cross-compile from a modern machine. Generate the C source, then transfer and compile on the target:

# On your modern machine: generate C source
mix --backend c --emit-ir -o game.c game.mix

# Transfer game.c and lib/runtime.c to target machine
scp game.c lib/runtime.c user@octane2:~/

# On the target: compile natively
cc -64 -O2 game.c runtime.c -o game -lmshell
Tip: The generated .c file and runtime.c are the only files needed on the target machine. No MIX compiler installation required.

05LLVM vs C Backend

Both backends produce identical program behavior — all tests pass with matching output. The differences are in compilation:

AspectLLVM BackendC Backend
SpeedFast default native pathDepends on target cc
Optimizationclang/LLVM via -O<level>target cc via -O<level>
Debug infoDWARF via --debugcc -g via --debug
DependenciesRequires clang; WebAssembly targets need WASI clang or EmscriptenOnly needs a C compiler
PortabilityNative clang targets plus wasm32 and wasm-browserAny platform with C99
Output qualityOptimized machine codeReadable C source
When to use which: Use the LLVM backend (default) for development on supported native and WebAssembly targets. Use the C backend when targeting unsupported architectures, when a C compiler is the only viable toolchain, or when you want to inspect the generated code.
